Lab #4: One Task is (almost) Never Enough

In this one-week exercise, you will put the code that you have developed in previous exercises into a real-time scheduler and test its performance. The scheduler allows your program to perform multitasking; you’ll need multitasking to be able to control the positions of multiple components for your term project. You’ve written code which can drive a motor, measure its position, run a generic controller, and display the results. Now it’s time to drive two devices simultaneously.

The Project

  • Create a new GitHub™ repository for this exercise as usual.

  • Download the file basic_tasks.py from the /examples folder of the ME405 support repository at https://github.com/spluttflob/ME405-Support. It’s easiest to download a .zip file using the Big Green Button, open the file in an archive manager, and extract those files into your src directory…though using Git to clone a copy of the repository is the cool way.

  • Check that you can import cotask and import task_share on your MicroPython board. Older versions of ME405 MicroPython will require you to download the files cotask.py and taskshare.py, then copy those files to the Nucleo.

  • Run basic_tasks.py and verify that it works. It should run, printing a counter being sent from one task to another, until you press Ctrl-C in the serial port terminal; then it should print tables of task and share information.

  • Rename basic_tasks.py as main.py. Use the example of a task function in main.py and what you’ve learned in lecture to write a task which runs the closed-loop motor controller you have previously written. Hook up a motor from your tub and run this task using the cotask.py scheduler with its task timing at around 10 ms.

  • Run the motor task on a motor with a flywheel, printing results and plotting step response graphs as you did in the previous exercise. Run the task at a slower and slower rate until the controller’s performance is noticeably worsened as shown on a step response plot. Record the slowest rate at which the performance is not significantly worse than when running the controller at a fast rate; this will help you choose a good run rate for the motor control task – it should be a bit faster than the slowest rate which works for a factor of safety. Save copies of the step response plots for the slowest rate at which the response is good and for a rate at which the response isn’t as good. Optional: Plot several responses on one set of axes, with a legend showing the task run rate of each.

  • Make two tasks which run two motors under closed-loop control at the same time. Write a test program which moves your motors simultaneously through different distances and holds them at the desired positions, and use it to test your code as thoroughly as possible. You don’t have to make two plots simultaneously, though it’s certainly OK if you’d like to.

    Alternative: If your term project will use a motor with another actuator (solenoid, linear actuator, PWM controlled flywheels, etc.) which will be actively controlled at the same time as the motor, you may write a multitasking program which controls the motor as assigned above plus your second actuator instead of a second motor. There must be some continuous activity to control the second actuator; this assignment is all about your microcontroller doing two things at once, not just starting one device and then controlling the other.

Deliverables

  • Before the due date, copy your source and documentation links to Can’tvas and push the following to GitHub™:

    • A README.md file for this experiment which briefly explains what you did in this experiment and shows how well your two-motor control system works. There should be discussion of what happened when one motor control task was run too slowly and recommending a speed at which the task should be run for proper operation; these discussions should be supported by at least two step response plots.

    • Your Python source code. Both the MicroPython files which run on the Nucleo and the CPython file you run on the PC should be in the /src subdirectory.

    • The Doxygen documentation should be shown on the GitHub™ pages as usual.

  • On the due date, you may be asked to demonstrate your program in lab.
